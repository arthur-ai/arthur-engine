# Design: Image Evaluation in Continuous LLM-as-a-Judge Evals

Generated by /office-hours on 2026-05-19 Branch: guardrails-v2 Repo: arthur-ai/arthur-engine Status: APPROVED Mode: Startup (intrapreneurship) Supersedes: ibrahim-main-design-20260406-134946.md

---

## Problem Statement

Arthur customers running multimodal LLM applications (vision models, image-captioning pipelines, document understanding apps) send image-bearing traces via OTEL/OpenInference. Their continuous LLM-as-a-judge evals produce no signal on those spans — because the eval pipeline is text-only. A customer asked for this directly.

---

## What Images Look Like in the Trace (Today)

An OpenInference span carrying an image message arrives as:

```
attributes.llm.input_messages.0.message.role = "user"
attributes.llm.input_messages.0.message.contents.0.message_content.type = "image"
attributes.llm.input_messages.0.message.contents.0.message_content.image.url = "https://..."
attributes.llm.input_messages.0.message.contents.1.message_content.type = "text"
attributes.llm.input_messages.0.message.contents.1.message_content.text = "Describe this chart"
```

After `SpanNormalizationService.normalize_span_to_nested_dict()`, this lands in `DatabaseSpan.raw_data` fully intact. The transform executor can extract the image URL with a path like:

```
attributes.llm.input_messages[0].message.contents[0].message_content.image.url
```

That extraction already works today — the result is a `str`. The problem is what happens *after* extraction.

---

## Premises

1. Images are sent via OTEL/OpenInference, not via a new upload endpoint. The ingestion path does not need to change.
2. `DatabaseSpan.raw_data` (JSONB) already stores image URLs from normalized spans.
3. The transform executor already extracts image URLs as strings — no changes needed there.
4. The LLM evaluator builds text-only LLM calls. It needs to build multimodal calls when any variable is an image URL.
5. `ChatCompletionService` already supports list-based `OpenAIMessage.content` (the code branches on `isinstance(message.content, list)`). The runtime plumbing for multimodal messages exists; it just isn't fed image content items yet.
6. The LLM model must support vision. Not all configured endpoints do. This must be validated or surfaced clearly, not silently fail.

---

## Approaches Considered

### Approach A: Declare image variables in eval definition (chosen)

```
Effort:  S
Risk:    Low
```

Add `image_variables: list[str] = []` to `CreateEvalRequest` and `DatabaseLLMEval`. Eval author declares which variable names are images. LLMEvaluator builds a multimodal message: the text prompt handles `{{text_vars}}` via Jinja2 as today; image variables are injected as `{"type": "image_url", "image_url": {"url": "..."}}` content items appended to the same user message.

- Pros: explicit, no auto-detect magic, backward-compatible, zero changes to transform layer or ingestion layer, one nullable column in `llm_evals` table
- Cons: eval author must manually declare image vars; no enforcement that the declared variable actually contains a URL at runtime

### Approach B: Type annotation in transform-to-eval mapping

```
Effort:  M
Risk:    Med
```

Add `variable_type: 'text' | 'image'` to `ContinuousEvalTransformVariableMapping`. The image annotation lives at the wiring layer (mapping), not the eval layer.

- Pros: same eval definition can be text in one task, image in another
- Cons: type annotation is further from the eval prompt, harder to reason about; requires changes to the mapping table and all mapping UI; the eval itself still needs to know which variables are images to build the message correctly

### Approach C: Auto-detect image URLs in resolved\_variables

```
Effort:  XS
Risk:    High
```

LLMEvaluator auto-detects if a variable value is a data URI or image URL.

- Pros: zero schema changes
- Cons: silently breaks on signed S3/GCS URLs (no file extension), ambiguous on text variables that happen to be URLs; non-deterministic behavior

---

## Recommended Approach: A

The eval definition is the right place to declare intent. The eval prompt already declares which variables it uses via `{{var_name}}` syntax. Declaring image variables alongside that is natural and keeps the type information co-located with the eval.

### 2\. Variable pipeline — full trace

The eval pipeline has several layers that all need to handle `image_variables` correctly:

**`src/repositories/llm_evaluator.py` — `LLMEvaluator.run()`:** Image variable names must be passed through to `LLMEvalsRepository.run_llm_eval()`. The evaluator currently builds `BaseCompletionRequest(variables=variables)` where `variables` comes from all `resolved_variables`. Change: filter image variables OUT of the `VariableTemplateValue` list (they are not Jinja2 template vars), and pass them separately:

```py
image_var_names = set(llm_eval.image_variables or [])
text_variables = [
    VariableTemplateValue(name=k, value=v)
    for k, v in resolved_variables.items()
    if k not in image_var_names
]
image_variables_dict = {k: v for k, v in resolved_variables.items() if k in image_var_names}
completion_request = BaseCompletionRequest(variables=text_variables)
llm_result = self._repo.run_llm_eval(
    ...,
    completion_request=completion_request,
    image_variables=image_variables_dict,  # NEW param
)
```

**`src/repositories/llm_evals_repository.py` — `run_llm_eval()` signature:** Add `image_variables: dict[str, str] | None = None` to the method signature and pass it down to `from_llm_eval_to_agentic_prompt()`:

```py
def run_llm_eval(
    self,
    task_id: str,
    eval_name: str,
    version: str,
    completion_request: Optional[BaseCompletionRequest] = None,
    image_variables: dict[str, str] | None = None,  # NEW
) -> ...:
    ...
    prompt = self.from_llm_eval_to_agentic_prompt(llm_eval, image_variables=image_variables)
    ...
```

**`src/services/continuous_eval/continuous_eval_queue_service.py` — expected\_vars check:** Lines 262-276 do `mapped_eval_vars != expected_vars` where `expected_vars` comes from `evaluator.get_eval_variables()`. This check will SKIP all multimodal evals because image variable names are in `resolved_variables` but NOT in the Jinja2-parsed template vars. Fix: `get_eval_variables()` in `LLMEvaluator` must return `llm_eval.variables + llm_eval.image_variables`. Update `LLMEvaluator.get_eval_variables()` accordingly.

**`src/repositories/llm_evals_repository.py` — `from_llm_eval_to_agentic_prompt()`:** This is where messages are built for the LLM call. The method constructs an `AgenticPrompt` with `messages: list[OpenAIMessage]`. To inject images, this method needs `image_variables` and must append `OpenAIMessageItem` objects of type `IMAGE_URL` to the user message's content list:

```py
def from_llm_eval_to_agentic_prompt(
    self,
    llm_eval: LLMEval,
    image_variables: dict[str, str] | None = None,  # NEW
) -> AgenticPrompt:
    image_vars = image_variables or {}
    if not image_vars:
        # existing text-only path, unchanged
        user_message = OpenAIMessage(role=MessageRole.USER, content=llm_eval.instructions)
    else:
        # Multimodal path: list-based content
        user_content = [
            OpenAIMessageItem(type=OpenAIMessageType.TEXT, text=llm_eval.instructions)
        ]
        for url in image_vars.values():
            user_content.append(
                OpenAIMessageItem(
                    type=OpenAIMessageType.IMAGE_URL,
                    image_url=ImageURL(url=url),
                )
            )
        user_message = OpenAIMessage(role=MessageRole.USER, content=user_content)
    ...
```

Use the existing `arthur_common` types (`OpenAIMessageItem`, `OpenAIMessageType`, `ImageURL`) — not raw dicts — so `model_dump(exclude_none=True)` produces the correct LiteLLM wire format. Verify: `OpenAIMessageItem(type=OpenAIMessageType.IMAGE_URL, image_url=ImageURL(url="https://...")).model_dump(exclude_none=True)` should produce `{"type": "image_url", "image_url": {"url": "https://..."}}`.

**Variable substitution** in `ChatCompletionService.replace_variables()` already skips non-TEXT content items (`chat_completion_service.py:122`), so image items are passed through to LiteLLM untouched. Image variable names are no longer in the Jinja2 variable map, so there is no risk of rendering URLs as Jinja2 templates.

### 3\. Vision model validation

`litellm.supports_vision(model, custom_llm_provider)` uses a static cost-map lookup. For Azure, the model name stored in Arthur is the deployment name (e.g. `"gpt4o-vision-deploy"`), which is not in LiteLLM's map — `supports_vision` will return `False` for every Azure vision deployment. **Do not block on this check.** Treat it as a soft warning only:

```py
if image_vars:
    provider = llm_eval.model_provider.value  # e.g. "azure", "openai"
    vision_check = litellm.supports_vision(llm_eval.model_name, custom_llm_provider=provider)
    if not vision_check:
        logger.warning(
            "Eval '%s' uses image_variables but litellm.supports_vision returned False "
            "for model '%s' (provider: %s). This may be a false negative for Azure/custom "
            "deployments. Proceeding anyway.",
            llm_eval.name, llm_eval.model_name, provider,
        )
```

At eval creation time: do NOT return a 422 on `supports_vision=False`. Azure users would always be blocked. Instead, validate that `image_variables` names are a subset of the variables parsed from the instructions template. In `save_llm_item()`, after `_extract_variables_from_item(item)` returns `all_variables`:

```py
# all_variables = self._extract_variables_from_item(item)  ← already computed here
unknown_image_vars = set(item.image_variables) - set(all_variables)
if unknown_image_vars:
    raise HTTPException(
        status_code=422,
        detail=f"image_variables {sorted(unknown_image_vars)} not found in eval template variables",
    )
```

### 4\. UI — eval creation form

In `ui/src/components/evaluators/EvalFormModal.tsx` (or `EvalEditModal.tsx`):

- After the instructions text field, parse variable names client-side with a regex: `/\{\{\s*(\w+)\s*\}\}/g` applied to the instructions string. This mirrors what the backend's Jinja2 parser does and avoids a round-trip to detect variables.
- For each detected variable, render a MUI `Chip` or `ToggleButton` pair: "Text" / "Image URL".
- Variables toggled as "Image URL" go into `image_variables` in the API request body.
- No change needed to the continuous eval wiring / transform mapping screens.

## End-to-End Flow (after this change)

```
1. Customer's vision app runs: user sends image + text prompt to GPT-4o
2. LangChain/LlamaIndex instruments the call via OpenInference
3. Span arrives at POST /api/v1/traces (protobuf)
4. SpanNormalizationService normalizes it → raw_data stores message_content.image.url
5. ContinuousEvalQueueService triggers for this trace
6. execute_transform extracts:
     input_image = "https://cdn.example.com/chart.png"  (image variable)
     response    = "The chart shows Q3 revenue of $2.4M" (text variable)
7. LLMEvaluator.run() → run_llm_eval() → from_llm_eval_to_agentic_prompt(image_variables=...)
8. Multimodal message: text prompt with {{response}} + image content item for input_image
9. LiteLLM calls GPT-4o with vision payload
10. Eval result (pass/fail + reason) stored as DatabaseAgenticAnnotation
```

## Open Questions

1. **Base64 images**: If the client sends a base64 data URI instead of a URL (some OpenInference instrumentation does this), does LiteLLM pass it through correctly? Test this. OpenAI accepts `data:image/jpeg;base64,...` in `image_url.url`.

2. **Image URL expiry**: If the image URL is a short-lived pre-signed URL and the continuous eval is delayed (jobs queue under load), the URL may expire before the LLM call. No solution for this in v1 — document it. If it becomes a real issue, the storage design from `MULTIMEDIA_DESIGN.md` (now deprecated) is the v2 path.

3. **Multiple images per span**: The design supports multiple image variables per eval. But what if a single span has 10 image messages? Extracting them requires 10 separate transform paths (one per index). Consider a wildcard extraction like `attributes.llm.input_messages[*].message.contents[*].message_content.image.url` — but the current transform executor uses first-match semantics. This is a follow-up.

4. **Cost tracking**: Multimodal LLM calls cost significantly more than text calls. The existing `LLMTokenConsumption` tracking should still work (LiteLLM returns token counts for vision calls), but the cost-per-token for image tokens may differ. Verify the cost columns in `DatabaseMetricResult` capture this correctly.

5. **`MULTIMEDIA_DESIGN.md`**: The preliminary design doc is now superseded. It describes a more complex path that would apply only if customers need to POST images directly (not via traces). Rename it or add a note at the top that the trace-path design is the current direction.

## The Assignment

Before writing any code: find one real span in staging that contains an image (`message_content.type = "image"`). Query `raw_data` directly in Postgres. Verify the path `attributes.llm.input_messages[0].message.contents[0].message_content.image.url` resolves to a real URL. That single query tells you whether the ingestion path is actually working end-to-end — or if there's a gap in `SpanNormalizationService` to fix first.

```sql
SELECT id, raw_data
FROM spans
WHERE raw_data::text ILIKE '%"type": "image"%'
LIMIT 5;
```

If that returns rows, the ingestion path is clean and you can start the eval changes. If it returns nothing, check `SpanNormalizationService._handle_message_contents()` — the image content items may be getting dropped or flattened.
